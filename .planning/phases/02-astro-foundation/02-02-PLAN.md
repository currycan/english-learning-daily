Append the `build-and-deploy` job to the end of `.github/workflows/daily-content.yml`. The complete job (per D-01, D-02, D-05, D-09; corrected from RESEARCH.md Pitfall 1):

```yaml
  build-and-deploy:
    needs: generate-content
    runs-on: ubuntu-latest
    permissions:
      pages: write
      id-token: write
    environment:
      name: github-pages
      url: ${{ steps.deploy.outputs.page_url }}
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '22'

      - name: Install website dependencies
        working-directory: ./website
        run: npm ci

      - name: Build Astro site
        working-directory: ./website
        run: npm run build

      - uses: actions/upload-pages-artifact@v3
        with:
          path: ./website/dist

      - uses: actions/deploy-pages@v4
        id: deploy
```

Key requirements:
- `needs: generate-content` — runs after content generation (per D-05, always rebuilds)
- `permissions: pages: write, id-token: write` — required for GitHub Pages deployment (per D-01)
- `environment: name: github-pages` — required for deploy-pages action (per D-01)
- `working-directory: ./website` on npm ci and npm run build — isolation per D-09
- `npm ci` (NOT `npm install`) — reproducible CI builds, requires committed package-lock.json
- `actions/checkout@v4` — MUST be present even though generate-content already checked out; each job gets a fresh runner (RESEARCH.md Pitfall 1)
- `actions/upload-pages-artifact@v3` with `path: ./website/dist` — packages the build output (per D-01)
- `actions/deploy-pages@v4` — deploys to GitHub Pages (per D-01, D-02 — no gh-pages branch)
- Keep existing `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true` env var at workflow level (per existing pattern in the workflow)

Do NOT modify the existing `generate-content` job or any other part of the workflow. Only append the new job at the end.

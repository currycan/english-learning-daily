## Recommended Stack

### Core Technologies

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| Astro | 6.x (current: 6.0.8) | Static site generator and build tool | Astro 6 went stable in March 2026 — it is the current supported version, not a pre-release. Breaking changes from 5 are minimal for content sites (removes `Astro.glob()`, upgrades to Zod 4, requires Node 22). New projects should use 6.x. Content Collections API is stable and unchanged. |
| Tailwind CSS | 4.x | Utility-first CSS for mobile-first design | Native Tailwind 4 support in Astro 5.2+ via `@tailwindcss/vite` plugin. The legacy `@astrojs/tailwind` integration is deprecated. No `tailwind.config.js` file in v4 — configuration lives in `src/styles/global.css`. Mobile-first breakpoints (`sm:`, `md:`, `lg:`) are Tailwind's default orientation. |
| date-fns | 4.x | Build-time date math for calendar grid | Tree-shakeable, zero client-side bytes. Runs only in Astro's build-time script context. Confirmed pattern for Astro calendar implementation (build-time `getStaticPaths`, no runtime). |
| TypeScript | 5.x (Astro-bundled) | Type checking across `.astro` components | Astro is TypeScript-first and ships its own TS toolchain — no separate `typescript` install needed. Use the "base" tsconfig preset to avoid over-engineering for a small project. |

|---------|---------|---------|-------------|

## Required: astro.config.mjs for GitHub Pages

```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import tailwindcss from '@tailwindcss/vite';

export default defineConfig({
  // Full URL of the deployed site — required for canonical links
  site: 'https://<username>.github.io',

  // Set to '/study-all' if repo name is study-all and not a root user pages repo
  // Omit or set to '/' if using a custom domain
  base: '/study-all',

  output: 'static', // default; explicit for clarity

  vite: {
    plugins: [tailwindcss()],
  },
});
```

Rules:
- `site` must be the full URL — used for sitemaps and canonical meta tags
- `base` must match the repository name exactly for subpath deployments
- All internal links must use Astro's `${import.meta.env.BASE_URL}` prefix or the Astro `<a href={base + '/lesson/' + id}>` pattern
- `public/.nojekyll` file is required to prevent GitHub Pages from processing the output with Jekyll

## Content Collection Approach: glob() Loader (Content Collections API)

**Use Astro Content Collections with the `glob()` loader — not `import.meta.glob()`, not the deprecated `Astro.glob()`.**

The existing `content/*.md` files have no YAML frontmatter. The `glob()` loader handles this via `.passthrough()` on the Zod schema:

```typescript
// src/content.config.ts  (note: NOT src/content/config.ts — Astro 5+ location)
import { defineCollection, z } from 'astro:content';
import { glob } from 'astro/loaders';

const lessons = defineCollection({
  loader: glob({ pattern: '**/*.md', base: './content' }),
  schema: z.object({}).passthrough(),
});

export const collections = { lessons };
```

Each entry's `id` is the filename without extension: `'2026-03-24'`. This is the natural date-based routing key.

**Section parsing** — split the raw Markdown body on `## ` headers at build time:

```typescript
// In a .astro page component (runs at build time)
const entry = await getEntry('lessons', '2026-03-24');
const sections = entry.body.split(/\n(?=## )/);
// sections[0] = "## 📖 文章 / Article\n..."
// sections[1] = "## 📚 词汇 / Vocabulary\n..."
// sections[2] = "## 🔗 表达 / Chunking Expressions\n..."
// sections[3] = "## ❓ 理解 / Comprehension\n..."
```

Pass each section string to Astro's built-in `<Content />` component or render via `marked` for HTML. This requires no remark/rehype plugin customization — simple string splitting on the fixed 4-section format is sufficient and has zero runtime cost.

**Comparison:**

| Approach | Handles No-Frontmatter | `getCollection()` API | Type Safety | Deprecated |
|----------|----------------------|----------------------|-------------|------------|
| `Astro.glob()` | Yes | No | No | Yes (Astro 6) |
| `import.meta.glob()` | Yes | No | No | No, but manual |
| Content Collections `glob()` loader | Yes (`.passthrough()`) | Yes | Yes | No |

## CSS: Tailwind CSS 4 (not plain CSS, not CSS Modules)

Tailwind CSS 4 via the `@tailwindcss/vite` plugin is the recommended approach for Astro 5.2+ and Astro 6.

Plain CSS is viable for 5 components but requires manual implementation of:
- Dark mode (`prefers-color-scheme` + class toggle logic)
- Responsive breakpoints (manual `@media` queries)
- Typography scale for comfortable mobile reading

Tailwind provides all three as utility classes. The dark mode implementation is a standard pattern:

```css
/* src/styles/global.css */
@import "tailwindcss";

/* Enable class-based dark mode (for manual toggle + system default) */
@custom-variant dark (&:where(.dark, .dark *));
```

```javascript
// Inline <script> in Layout.astro — dark mode initialization
const theme = localStorage.getItem('theme');
const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
if (theme === 'dark' || (!theme && prefersDark)) {
  document.documentElement.classList.add('dark');
}
```

Toggle button updates `localStorage` and toggles the `dark` class on `<html>`. This runs client-side (~10 lines of vanilla JS) with no framework dependency.

## Calendar: Pure JS + date-fns (build-time HTML, client-side read indicators)

No calendar library is needed. The calendar grid is rendered as static HTML at build time.

**Build-time (in .astro file):**
```typescript
import { eachDayOfInterval, startOfMonth, endOfMonth, format, getDay } from 'date-fns';
// Generate grid cells for a given month
const days = eachDayOfInterval({ start: startOfMonth(month), end: endOfMonth(month) });
```

**Client-side (vanilla script):**
```javascript
// Read localStorage and mark read cells
const read = JSON.parse(localStorage.getItem('read_lessons') || '[]');
document.querySelectorAll('[data-date]').forEach(cell => {
  if (read.includes(cell.dataset.date)) cell.classList.add('is-read');
});
```

Total client-side JavaScript: ~15 lines. Zero framework dependencies.

Avoid: `react-calendar`, `FullCalendar`, `Vanilla Calendar Pro` — all add unnecessary bundle weight for a static read-only archive grid.

## GitHub Actions Deployment

**Use `withastro/action@v3` + `actions/deploy-pages@v4`.** Do not use `peaceiris/actions-gh-pages`.

Rationale: The `withastro/action` + `deploy-pages` pattern uses the native GitHub Pages artifact API, which works with the auto-provided `GITHUB_TOKEN`. No Personal Access Token or deploy key required.

The `peaceiris/actions-gh-pages` alternative writes to a `gh-pages` branch and requires a PAT or deploy key — unnecessary complexity for this project.

**Recommended workflow** (new file: `.github/workflows/deploy-website.yml`):

```yaml
name: Deploy website

on:
  push:
    branches: [master]
    paths:
      - 'content/**'   # Triggers when daily-content.yml commits new .md file
      - 'src/**'
      - 'astro.config.mjs'
      - 'package.json'
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: withastro/action@v3
        # node-version: defaults to 24 (matches FORCE_JAVASCRIPT_ACTIONS_TO_NODE24 in existing workflow)
        # package-manager: auto-detected from lockfile

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

**Integration with `daily-content.yml`:** The `push: paths: content/**` trigger fires automatically when the existing Python pipeline commits a new `.md` file. No changes to `daily-content.yml` are required — the two workflows are decoupled.

## TypeScript: Yes, "base" Preset

Astro ships TypeScript tooling. Use it with the "base" preset (not "strict"):

```json
// tsconfig.json
{
  "extends": "astro/tsconfigs/base"
}
```

"base" avoids strict null checks that slow down small-project development while still catching type errors in the content loading, date parsing, and section splitting logic.

No separate `typescript` npm install required.

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
|-------------|-------------|-------------------------|
| Astro 6.x | Next.js | When the project needs SSR, API routes, or React ecosystem deeply integrated |
| Tailwind CSS 4 | Plain CSS | For a site with fewer than 5 components and no dark mode toggle requirement |
| Tailwind CSS 4 | CSS Modules | In large teams requiring strict style isolation at component level |
| date-fns (build-time) | Vanilla Calendar Pro | When you need interactive date-picking (not needed here — read-only archive grid) |
| `withastro/action@v3` | `peaceiris/actions-gh-pages` | When you need a `gh-pages` branch output for legacy tooling; not justified here |
| Content Collections `glob()` loader | `import.meta.glob()` | When files have completely heterogeneous structure and `getCollection()` isn't needed |
| Astro 6.x | Astro 5.x | Pin to 5.x only if Zod 4 migration causes an unexpected issue; otherwise prefer 6.x |

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| `@astrojs/tailwind` | Deprecated in Astro 5.2+; removed path is the Vite plugin | `@tailwindcss/vite` Vite plugin |
| `Astro.glob()` | Deprecated in Astro 5, removed in Astro 6 | Content Collections `glob()` loader |
| React / Vue / Svelte components | Dark mode toggle + localStorage read tracking need ~30 lines of vanilla JS total; no framework warranted | Vanilla `<script>` tags in `.astro` files |
| `moment.js` or `luxon` | Heavy; `date-fns` is tree-shakeable and sufficient for build-time date math | `date-fns` 4.x |
| YAML frontmatter in content files | Existing Python pipeline generates files without frontmatter; adding it would require modifying the pipeline | Schema `.passthrough()` on the glob loader |
| `remark`/`rehype` plugins for section splitting | Overkill; the four-section format is fixed and a single `String.split(/\n(?=## )/)` is sufficient | String splitting in build-time component logic |
| `public/.nojekyll` omission | GitHub Pages will try to process the output with Jekyll and fail silently | Add empty `public/.nojekyll` file |
